Day 1 — 29/06

What was done

Bug: duplicate modals
The /add-office flow opened two identical modals. Cause: views_push on a non-parent view in Slack's view stack.
Fix: response_action="update"
Replaced the current modal with response_action="update" instead of pushing/opening new views. The rooms modal now replaces the office modal.
Removed: Owner field
Removed the owner_slack_id field from the form. Config data (timezone, hours, city) now travels via private_metadata.
Analysis
Root cause

Why the modals were duplicating

views_push
Slack requires the current view to have a parent in the stack. If the view has already been replaced or is root, views_push fails or opens a duplicate.
Early DB save
The office was created in the DB on the first submit. If the user closed the second modal, an orphan record remained. If they retried, a duplicate was created.
Intermediate attempt: views_open
views_open doesn't work when a modal is already open from the same command. It created a loop — the original modal kept reopening.
Solution: response_action="update"
Replaces the current view with a new one — no stack involved. This is the correct approach for multi-step flows in Slack.
Deferred creation
Office data is passed to the second modal via private_metadata. The office is only created when the user completes room configuration.
Owner removed
Owner field was unnecessary in the current flow. Simplifies the UI and reduces friction in office creation.
Implementation detail

How it works now

# add_office.py — New flow # 1. First modal: collects office data # No DB write — validation only # 2. Submit → response_action="update" ack( response_action="update", view=_build_rooms_view( office_name=..., # passed via num_rooms=..., # parameters, not timezone=..., # via database ), ) # 3. Second modal: configure rooms # DB write happens here only conn = get_connection() office = OfficeRepository(conn).create( name=office_name, timezone=timezone, business_hours_start=..., business_hours_end=..., city=city, )
private_metadata payload
{ "office_name": "Roma", "num_rooms": 3, "timezone": "Europe/Rome", "business_hours_start": "09:00", "business_hours_end": "18:00", "city": "Roma" }

Two problems, two ideas

The desk menu was broken
Slack allows at most 100 options in a dropdown menu. With offices of 200–300 workstations, more than half the desks couldn't be selected.
The idea
Load options on demand, filtered by office and time slot — no limits.
Desks are just numbers
A user books "desk #47" without knowing where it is. The idea: draw the office floor plan and click directly on a workstation.
The idea
Visual editor with Konva.js: admins drag desks onto a map. Users then just click the floor plan to book.
